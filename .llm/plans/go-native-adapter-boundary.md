# Implementation Plan: Go-Native Adapter Boundary

## Context

Spec: `.llm/features/go-native-adapter-boundary.md`

ADR: `docs/engineering/decisions/0005-generated-go-emission-boundary.md`

Compiler/runtime boundary:
`docs/engineering/decisions/0006-gowdk-compiler-and-runtime-boundary.md`

Language source of truth: `.llm/features/deep-go-package-integration.md`

GOWDK should stay compile-first, but Go developers should not have to trust a
large generated codebase that appears to own their backend. The right boundary
is small adapter generation: `.gwdk` declarations become route metadata, user
handlers stay normal Go, and generated packages only bridge `net/http`,
GOWDK runtime contracts, and user feature packages.

## Assumptions

- Generated Go remains necessary for embeddable app binaries and adapter glue.
- User application logic is always hand-written Go outside generated output.
- The first backend handler signatures stay narrow while the adapter boundary is
  stabilized.
- Existing generated app behavior should keep working during the refactor.
- This plan must not define a competing `.gwdk` syntax. Package-first syntax
  and migration are owned by the deep package integration plan.

## Proposed Changes

- Define the generated-code policy in docs.
  - Generated Go is compiler-owned adapter glue.
  - User handlers, domain logic, auth, storage, and service code are never
    generated by GOWDK.
  - Generated Go is emitted from Go ASTs. Raw Go templates are migration debt,
    not an accepted path for new generated Go.
  - Generated Go must not be assembled with hardcoded line writing,
    `WriteString` chains, token concatenation, or source snippets unless a
    temporary exception is strictly necessary and documented at the call site.

- Lock the feature-bound backend model.
  - Keep same-package action/API discovery as the first Go-native backend path.
  - Emit binding metadata for every action/API declaration.
  - Import user feature packages only for valid bound handlers.
  - Return clear `501` responses for missing or unsupported handlers.

- Introduce a typed adapter IR plus full Go AST emission model.
  - Add an internal route adapter IR that represents imports, route predicates,
    request decoding, handler calls, and response writing.
  - Generate action/API/SSR dispatch from that IR through `go/ast`.
  - Generate file shells, import declarations, embedded assets, and top-level
    runtime wiring through `go/ast`.
  - Print and validate generated Go with standard Go tooling before writing.

- Remove raw generated-Go templates in phases.
  - First move action/API handler bodies to AST generation.
  - Then move backend proxy and SSR dispatch through the same AST-backed route
    model.
  - Then move remaining generated Go file shells to AST generation.

- Add guardrails.
  - Tests assert generated code does not define user handler functions.
  - Tests assert missing handlers do not create imports or symbol references.
  - Tests assert generated source formats successfully before write.
  - Tests and review check that generated Go paths use AST builders instead of
    line-by-line source writing.
  - Docs and examples refer to generated code as adapter glue.

## Data And API Impact

- `.gwdk` syntax breaks are owned by the package integration plan. This plan
  consumes the normalized route and binding model after parsing.
- Public config and CLI can keep the feature-bound backend fields already used
  by the backend integration slice.
- Manifest/build report metadata should continue exposing backend binding status.
- Generated adapter source layout may change, but generated output remains an
  implementation artifact.

## Implementation Phases

### Phase 1: Policy And Current Slice Alignment

- Add this spec and plan.
- Update ADR 0005 to point at the concrete adapter boundary.
- Finish docs that still describe generated apps as unable to call user
  action/API handlers.
- Ensure the login example demonstrates normal Go handlers imported by generated
  adapters.

### Phase 2: Stabilize Binding Contracts

- Add focused tests for action/API binding discovery.
- Cache Go package inspection by feature directory.
- Verify bound, missing, and unsupported statuses in manifest JSON, route output,
  and build reports.
- Keep unsupported signatures non-fatal until typed handler expansion is planned.

### Phase 3: Typed Adapter IR

- Add an internal adapter model for:
  - required imports,
  - route match conditions,
  - request method and path handling,
  - form/body decoding steps,
  - user handler call targets,
  - response writer behavior,
  - fallback `501` behavior.
- Convert action and API generation to consume this model.
- Keep generated code deterministic by sorting imports and routes.

### Phase 4: Full Go AST Emission

- Replace handler-body string assembly with `go/ast` construction.
- Replace generated import blocks, route registration, backend proxy, SSR
  dispatch, and remaining generated Go file shells with `go/ast` construction.
- Use `go/printer` or equivalent standard AST printing, then `go/format` as the
  hard validation gate before writing files.
- Keep any helper layer thin and typed; it should make AST construction readable
  without becoming a second source language.
- Add golden or structural tests for generated source shape.

### Phase 5: Unify One-Binary And Split-Binary Paths

- Drive frontend proxy, backend-only app, and one-binary app from the same route
  adapter IR.
- Keep split mode optional; one generated binary remains the default product
  path.
- Add end-to-end tests for one-binary and split-binary action/API execution.

### Phase 6: Typed User Contracts

- Plan typed action input structs after the adapter boundary is stable.
- Add typed decoder generation without generating user structs or user handler
  bodies.
- Add CSRF wiring as runtime adapter behavior, not user logic generation.

## Tests

- Unit:
  - backend binding discovery for bound, missing, and unsupported handlers.
  - adapter IR construction for action, API, backend proxy, and SSR routes.
  - import alias collision handling.
  - generated source formatting failure paths.

- Integration:
  - generated one-binary app calls normal Go action/API handlers.
  - generated split backend app calls the same handlers.
  - generated frontend proxy forwards backend action/API endpoints.
  - missing handlers return clear `501` responses.
  - generated Go files are produced from AST builders and format successfully.

- End-to-end:
  - login example sign-in, session API, dashboard, and logout in one-binary mode.
  - login example sign-in/session flow in split-binary mode.

- Manual:
  - inspect generated adapter source for the login example and confirm it only
    contains glue code.
  - inspect `gowdk routes` and build report binding metadata.

## Rollback Plan

- Keep existing generated app shell output while replacing route emission one
  route kind at a time.
- If AST adapter emission causes churn, leave the current formatter gate in place
  and revert only the new AST emitter package.
- If split-binary unification regresses, keep one-binary generation and restore
  split frontend/backend generation to its previous direct implementation.
- Do not revert the product boundary docs unless the project intentionally
  decides to generate user application logic.

## Risks

- A full `go/ast` rewrite can become verbose. Keep helper functions small,
  boring, and close to Go syntax instead of inventing a large codegen DSL.
- Import cycles are possible if user feature packages import generated output.
  Document that generated packages are leaf adapters.
- Binding discovery can get slow without caching.
- Over-constraining generated source shape can make legitimate compiler changes
  noisy. Favor structural assertions over brittle full-file snapshots where
  possible.
